## 一、校历 / 赛季规则（已实现，推荐默认）

**结论：一学期 = 一个赛季，全年两个赛季，重置点对齐两个真实开学日。**

| 赛季 | 时间区间 | 赛季键 | 说明 |
|---|---|---|---|
| 秋季赛季 | **9/1 – 次年 1/31** | `<起始年>-A` | 秋季学期。1 月仍属上一年的秋季赛季。 |
| 春季赛季 | **2/1 – 8/31** | `<年>-S` | 春季学期 **+ 暑假**。暑假并入春季，学生整个假期保住春季段位（软地板按赛季，跨赛季才重置）。 |

- **只有两个边界月可调**：`SPRING_START_MONTH = 2`、`AUTUMN_START_MONTH = 9`（`periods.ts`）。若贵校校历不同（如 9/1 改 8/25、2/1 改 2/15 之类的"周"级差异），改这两个常量即可——但全国榜需要**统一**一套，建议按月对齐而非按校精确到天。
- **为什么对齐"开学日"而非"期末考"**：开学日是对学生最有意义、全国最一致的重置点；段位在新学期清零重燃斗志，符合 decision 7"赛季内不掉段"的初衷。
- **赛季榜已上线**：每次推进回合会写入 周/月/**赛季** 三个榜。UI 段位卡可用 `seasonLabel("2026-S") → "2026 春季赛季"`。

**示例**（`semesterKey`）：2026-06-01 → `2026-S`；2026-09-01 → `2026-A`；2027-01-10 → `2026-A`。

**❓需你确认**：(a) 接受 9/1 与 2/1 两个边界？(b) 暑假并入春季（而非单独"暑期赛"）是否 OK？

## 二、学习进度接入评分（提案，待选型）

战力公式里 `学习` 占 **15% 权重**，目前因无追踪恒为 0（透明面板里诚实显示为"可提升的杠杆"）。要接入需新建一个**按用户的学习完成度子系统**，并先定"怎样算完成一课"。

### 三种"完成"判定模型

| 方案 | 怎样算完成 | 严谨度 | 需要的额外内容 | 防刷 |
|---|---|---|---|---|
| **A 浏览即完成** | 打开课程模块即标记完成 | 低 | 无（立即可上线） | 弱（可秒点） |
| **B 测验通过**（推荐终态） | 每课 3 题小测，答对 ≥2 即完成 | 高 | 需为每个学习模块编写测验题 | 强 |
| **C 时长+读完** | 停留 ≥N 秒且滚动到底 | 中 | 无 | 中（可挂机） |

**推荐：分两步走 A → B。** 先上 **A**（零额外内容，立刻让"学习"维度非 0、形成激励闭环），随后按模块补测验题逐步升级到 **B**（真正奖励"学懂了"，与反 YOLO 的产品气质一致）。

### 评分接入（已就位，无需改公式）
- `learningCompleted` = 用户已完成模块数；`learningTotal` = `learningModules.length`。
- `learning` 分量 = `completed / total`（`power-score.ts` 已实现）。
- `recomputePowerForUser` 取 `getLearningProgress(userId)` → 传入 `runToPowerInput`。

### 需要新建的部分（A 方案工作量）
1. **迁移 0012** + 表 `learning_progress(user_id, module_id, completed_at, [score])`，唯一键 `(user_id, module_id)`。
2. store/repo：`markModuleComplete(userId, moduleId)`、`getLearningProgress(userId) → {completed,total}`（双写 + 内存兜底）。
3. API：`POST /api/learn/complete { moduleId }`（B 方案再加测验作答与校验）。
4. UI：`/learn` 模块页加"完成学习"按钮（B 方案为小测验）。
5. 接线：`recomputePowerForUser` 读取学习进度传入评分。

**❓需你确认**：(a) 选 **A 先上、后升 B** 还是直接 **B**？(b) 若 B：测验题由谁产出（你方/我方草拟）、每课题量与通过线？(c) `learningTotal` 用全部模块数，还是仅"必修"模块数？
