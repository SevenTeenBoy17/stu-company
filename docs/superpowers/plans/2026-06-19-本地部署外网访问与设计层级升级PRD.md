### D1 · 生产配置（env + serving）
强制 env(写进生产启动环境,**不要进 git**):
| env | 值 | 作用/缺失后果 |
|---|---|---|
| `NODE_ENV` | `production` | 开 secure cookie + 开 checkOrigin + 关内存兜底。**缺=CSRF 与安全 cookie 同时失效** |
| `APP_URL` | `https://你的隧道域名` | checkOrigin 同源白名单(`api-response.ts:53`)+ 邮件链接基址。**设错自己也被 403** |
| `SESSION_SECRET` | ≥32 强随机 | JWT 密钥,生产缺失启动即抛错 |
| `DATABASE_URL` | 本机 `localhost:5433` | 写路径持久化 |
| `ALLOW_MEMORY_FALLBACK` | **不要设**(=false) | 一旦 true,DB 抖动时登录会落到 store.ts 的公开口令超管(`repo.ts:1188`)——对外=P0 |
- serving:`npm run build` →(生产环境)`npm run start`。**绝不能用已在跑的 `npm run dev` 对外**(dev 会让 checkOrigin+secure cookie 全失效)。
- 验收:`NODE_ENV=production npm run build` 通过;`npm run start` 本机 `curl localhost:3000` 200。

### D2 · 公网隧道（Cloudflare Tunnel,推荐）
```powershell
winget install --id Cloudflare.cloudflared          # 1) 安装
cloudflared tunnel login                            # 2) 授权(需一个你拥有的域名)
cloudflared tunnel create brownzone                 # 3) 创建命名隧道(稳定 ID)
cloudflared tunnel route dns brownzone app.你的域名.com   # 4) 绑 DNS
# 5) config.yml: ingress → hostname app.你的域名.com → service http://localhost:3000
cloudflared tunnel run brownzone                    # 6) 前台验证
cloudflared service install                         # 7) 装成 Windows 服务(开机自启)
```
- 备选:**ngrok**(一行最快,但免费版域名临时、破坏 APP_URL 固定值,仅适合临时演示);**路由器端口转发+DDNS+Caddy**(暴露家庭 IP、需自管证书、攻击面大,不推荐)。
- 验收:外网另一台设备/手机流量打开 `https://app.你的域名.com`,能**注册新账号→登录→进 /student→数据落库刷新仍在**。

### D3 · 上线后硬性注意
PC+隧道常开(关睡眠);DB 只绑 localhost;`APP_URL`=隧道域名;`x-forwarded-for` 由 cloudflared 注入(限流键才可信,别在前面再叠会改写该头的不可信代理);单机单进程→限流/缓存有效,**切勿无 Redis 多实例**。

# Part 2 · 字体 / 字号 / 颜色 层级升级（抓得住重点、提留存）

## 2.1 根因(实测,三处系统级断层)
1. **字体根本没加载**:`layout.tsx` 只 `import globals.css`,之前为修生产构建把 `next/font/google` 删了(`progress.md:117`)→ 全部回落系统字体,设计稿假设的 Noto Sans SC/Inter 等没渲染,**字重对比先天打折**。
2. **`font-black`(900) 滥用**:学生组件命中 **395 次/19 文件**——标题、正文、徽章、按钮、百分比全是 900,weight 不再承载层级=flat。
3. **硬编码颜色绕过语义层**:`text-slate-*/amber-*/orange-*` 命中 **763 次/22 文件**,`--color-fg-*` 形同虚设;没有"唯一强调色";`risk-profile` 的 `bandClass` 四个风险档**全映射到同一个 `from-slate-950`**(`:37-42`)——四档视觉一样,区分信息丢失。
4. **缺"英雄数字"**:每屏第一数字(净值/总分/温度分)没有"明显大一档"的处理,KPI 全同级。
- 注:`money-text.tsx:32` 的 `font-extrabold tabular-nums`+红涨绿跌是全仓最对的一处,**保留**。最近一轮调暗过 AA 的语义文字色**只可在其上构建,不可回退**。

## 2.2 方案 A · 决定性 Type Scale（用 SIZE 拉层级,WEIGHT 收敛到 3 档 400/600/800）
| role token | size/行高 | weight | 用途 |
|---|---|---|---|
| `text-display-xl` | 44/50 | 800 | hero 大标题 H1 |
| `text-display-lg` | 34/40 | 700 | 区块大标 |
| **`text-hero-num`(新增)** | 40/44 | 800 + tabular | **每屏唯一英雄数字**(净值/总分/战力) |
| `text-h1` | 28/36 | 700 | 面板标题 |
| `text-h2` | 22/30 | 700 | 卡片标题 |
| `text-h3` | 18/26 | 600 | 小卡/列表项标题 |
| `text-body` | 15/24 | **400**(从 black 改回) | 标准正文 |
| `text-body-sm` | 13/20 | 400 | 辅助说明 |
| `text-caption` | 12/16 | 600 + 字距 | 标签/元数据(eyebrow 靠字距不靠 900) |
- **铁律**:一个卡片内最多 2 档字重(标题 700/800 + 正文 400);层级靠 size 不靠 weight;数字永远 tabular-nums,英雄数字升 `text-hero-num`。

## 2.3 方案 B · 色彩强调系统（保留红涨绿跌+琥珀,建立"唯一强调"）
| role | token/值 | 何时用 |
|---|---|---|
| 文字-强 | `text-fg-strong`(ink-800) | 标题、英雄数字 |
| 文字-默认 | `text-fg-default`(ink-900) | 正文 |
| 文字-次 | `text-fg-muted`(ink-600,已 AA) | 说明/caption |
| **唯一品牌强调** | `bg-brand`(amber)+ink-900 | **每屏 1 个主 CTA + 英雄数字容器**,其余按钮降级描边/幽灵 |
| 涨/正 | `text-up`(up-600) | 价格涨/正收益(**保留**) |
| 跌/负 | `text-down`(#107a3e) | 价格跌/负收益(**保留**) |
| 警示/危险/信息 | warning(#854d0e)/error/info(info-600) | 一致语义,已 AA |
- **三条规则**:① 每屏唯一品牌色焦点(深色反相面板从"每块都 slate-950"减到只剩 hero 一块);② **修风险四档同色**(defensive 蓝灰 / steady 绿向 / balanced 琥珀 / growth 红向,颜色本身=教学信息);③ 深底文字也收敛为 3 档透明度(标题/正文/caption)。
