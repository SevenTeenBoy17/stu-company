## 2. 结论速览

**代码健康度（实测，非估计）**：`npx tsc --noEmit` ✅ 0 错误 · `npm run lint` ✅ 0 警告 · `npm run test` ✅ **370 passed / 0 failed / 0 skipped**（52 个测试文件，Vitest 4.1.8，30.5s）。工程基础设施扎实，主干分支可编译可测试。

**但存在必须在对外推广前修复的问题：**

- **1 个 P0（可被未认证攻击者直接利用）**：邀请码用 `ilike` 匹配用户输入，`%`/`_` 通配符可让任何人自助注册成**教师**账号，并用一条 SQL 批量篡改所有邀请码。已对照源码核验为真。
- **3 个 P1 安全/鉴权**：邮箱同样用 `ilike` 解析破坏账号唯一性；注册限流键取自请求体邮箱形同虚设；缺少 CSP/HSTS 等安全响应头。
- **3 个 P1 战力榜刷分缺口**：确定性升值资产 + 无上限杠杆可零风险刷到最高段位；「什么都不做」基线高达 1110 战力；战力榜不校验赛季种子、不要求完赛，可「重玩选种 + 见好就收」刷榜。**这三条直接违背产品「评判决策质量而非运气（反 YOLO）」的核心教学承诺。**
- **多个 P1 后端性能**：每次读沙盘状态全表扫描 users+runs；单次推进回合 ~25 次串行 DB 往返；榜单全量加载后在应用层分页。在「中国用户 → us-east-2 数据库、5 秒往返」的真实链路下，这些是超时主因。
- **2 个 P1 前端**：操作按钮防双击失效（`void mutate` 使 `pending` 不覆盖请求期）；榜单 API 失败把已入榜学生打回入榜表单、可能覆盖隐私设置。

完整清单见 `01-问题汇总.md`。
