# 任务中心 盲盒升级 — 技术可行性 PUNCH-LIST (P0/P1/P2)

Reviewed against the live code: `student-quest-dashboard.tsx`, `cards.ts`, `content.ts`, `quests.ts`, `motion-system.ts`, `draw/route.ts`, `page.tsx`, `repo.ts`. Each item is concrete + actionable. **The four specs are largely sound and well-grounded, but they collide on the data model, contradict each other on rarity, and several "extend the existing component" claims understate the real surgery.**

## P0 — Blockers (must resolve before any implementation starts)

**P0-1 · `questCardDeck` is `as const` — every proposed field add is a type rewrite, not an extension.**
`content.ts:111` ends `] as const`, and `drawCard(deck: readonly QuestCard[], …)` plus `findDeckCard`/`page.tsx:31` (`card as QuestCard`) all consume it. The integration-map and gamification specs blithely "add `series`/`seriesNum`/`colorIdentity`/`tagline`/`mascotUrl`" to cards. Reality: you must (a) extend the `QuestCard` type in `cards.ts:3`, (b) add the fields to **all 12** literal entries or the `as const`-narrowed object stops satisfying `QuestCard`, and (c) keep them serializable (the card is JSON-returned from `draw/route.ts:106` and stored in DB `meta`). Decide field optionality up front: making them required breaks `findDeckCard` returns and the `page.tsx` cast until all 12 are filled. **Action: land the `cards.ts` type + all 12 deck entries as one atomic commit before touching any UI.**

**P0-2 · The four specs contradict each other on `legendary`. Pick one, in writing, now.**
- integration-map + asset-pipeline: **add** a 4th `legendary` tier (extend `QUEST_CARD_RARITY_WEIGHTS`, add 2 mascots #13–14, `rarityMeta.legendary`).
- motion-spec + a11y-spec: **explicitly forbid** it — "稀有度只有三档… legendary 一律映射为 epic", and `rarityMeta` is `Record<QuestCard["rarity"], …>` (a11y-spec relies on this exhaustiveness).
Shipping both halves = `rarityMeta` is missing a `legendary` key → TS error at `cards.ts` exhaustive `Record`, and `pickRarity` (`cards.ts:31`) silently includes/excludes it based on deck presence. **Action: product decision. If yes-legendary: it is its own scoped sub-task (type + weight + `rarityMeta` AA color + 2 art assets + every `Record<QuestCardRarity>` map updated). If no: strike all legendary content from integration-map/asset-pipeline. Do not start UI until resolved.**

**P0-3 · `drawCard` signature change (`effort` rarity-floor) is an API + determinism change, not a UI tweak.**
gamification §1.2 adds a 4th param `effort` to `drawCard` and an `effort` field to `StudentQuestItem`. But `StudentQuestItem` is built in `quests.ts:301` (10 hand-written quests, none have `effort`), and `draw/route.ts:91` calls `drawCard(questCardDeck, ownedCardIds, seed)` with no effort. Changing the seed-to-card mapping **retroactively changes which card every existing student already drew** — the route is idempotent via `existingCardForTrigger` (`draw/route.ts:73`) so *already-drawn* cards are safe, but any not-yet-drawn quest will now resolve differently, and the property-based determinism guard (`cards.test.ts`) will need new cases. **Action: gate behind a defaulted param (`effort = "standard"`) so the call site stays valid; add `effort` to all 10 quests in `quests.ts`; add a determinism test that the floor never *lowers* an existing rarity. Treat as a `finance_event_simulator`/`api_wirer` task, not `ui_implementer`.**

**P0-4 · "Extend `StudentQuestItem` with mascot fields" forces a `buildStudentQuestPayload` rewrite + breaks `claimQuestReward`.**
integration-map §1.2 adds `seriesId/mascotName/mascotColorHex/mascotImageUrl/rarity` to `StudentQuestItem`. That interface (`quests.ts:11`) is built in **two** places — `buildStudentQuestPayload` and re-built inside `claimQuestReward` (`quests.ts:513`) — and consumed by `draw/route.ts:64`. Every one of the 10 literal quest objects must gain the new fields or TS fails. Also note: a quest's `rarity` (proposed on the quest) and the drawn **card's** `rarity` are different concepts; the spec conflates them. The card you draw is `drawCard`-selected, not the quest's declared rarity. **Action: clarify that quest-level `mascot*`/`series` fields are presentation metadata only and MUST NOT feed `drawCard` rarity (which would break the anti-gacha weighted draw). Keep the draw deterministic and quest-rarity-independent.**

**P0-5 · `MascotRewardModal` / celebration modal has zero focus-management scaffolding today — and the existing `QuestDetailDialog` is the broken template it's told to copy.**
`QuestDetailDialog` (`student-quest-dashboard.tsx:671`) has **only** `role="dialog" aria-modal="true"` — no focus trap, no ESC, no return-focus, no `aria-labelledby`, no scroll-lock, no body-portal. The a11y-spec §3.2/§3.3 correctly flags this, but the integration-map and motion specs both say "reuse `QuestDetailDialog`'s mask structure" as if it's a solid base. It isn't. **Action: build a shared `useFocusTrap(ref, {onClose})` hook FIRST, retrofit `QuestDetailDialog`, then build the reward modal on it. The reward modal is net-new component work (~280 lines) with a hard a11y contract, not a restyle. Owner: `teen_ux_specialist` + `testing-accessibility-auditor` review.**

## P1 — High-risk, needed for fidelity (do after P0)

**P1-1 · GSAP scope/contextSafe expansion is correct in the motion-spec — but the new event handlers must NOT be created inside the existing single `useGSAP`.**
The current `useGSAP` (`:853`) returns `{ contextSafe }` and the two existing handlers (`animateQuestCard`, `toggleQuestFlip`) are created via `contextSafe` *outside* the callback (correct). The motion-spec's skeleton §9 is faithful. **Risk:** `pointermove` hover-tilt listeners and `IntersectionObserver`/`ScrollTrigger` (foil shimmer §6, grid stagger §7) need explicit teardown in the `useGSAP` cleanup return — `useGSAP` auto-reverts *tweens* in scope but does **not** remove your manually-added `addEventListener`/`observer`. The spec says this but it's the #1 leak source in React 19 StrictMode double-invoke. **Action: every listener/observer added must be in the `useGSAP` return cleanup; add an E2E or manual unmount check. ScrollTrigger also requires `gsap.registerPlugin(ScrollTrigger)` — currently only `useGSAP` is registered (`:39`); pick IntersectionObserver to avoid a new plugin dependency unless ScrollTrigger is already bundled.**

**P1-2 · Foil shimmer / idle glow are the ONLY allowed `repeat:-1` loops — enforce the reduced-motion + offscreen kill, and respect the no-blanket-`will-change` rule.**
Motion-spec §6 is correct that `ANIM-JANK-2` (globals.css) forbids resident `will-change` except for continuous loops. Two concrete gotchas the implementer will miss: (1) the current reduced-motion branch (`:855`) only resets `[data-quest-reveal]`, `[data-calendar-cell]`, `[data-quest-card-inner]` — any new `[data-reward-foil]`/glow loop must be added to that early-return guard or it runs under reduce-motion (WCAG 2.2.2 violation). (2) The collection grid can render up to 12 cards (`QuestCardCollection` 4-col); starting a `repeat:-1` foil on every rare+ card simultaneously is a real perf hit. **Action: start foil only on rare/epic, only when in viewport, `pause()` offscreen, and kill in cleanup; add the new selectors to the reduce-motion `gsap.set` reset list.**

**P1-3 · Count-up (§8) on the hero numbers conflicts with current inline `width:%` bars and the `data-motion-viz-bar` system.**
The hero "任务完成度 {completed}/{total}" (`:1078`) and progress bars use inline `style={{ width: '...%' }}` (`:1083`, `:1459`). The a11y-spec adds `role="progressbar"` + `aria-valuenow` (good). But motion-spec §8 count-up writes `el.textContent` on `[data-motion-number]` — **none of these hero numbers currently carry `data-motion-number`/`data-count-to`**, and they render as JSX expressions (`{a}/{b}`) split across multiple text nodes (`:1090-1092`). You can't `textContent`-animate a node that React owns and re-renders. **Action: wrap each animated number in a single dedicated span with `data-motion-number`, and ensure GSAP writes don't fight React re-renders (animate a proxy, write on `onUpdate`, and `killTweensOf` before re-run as the spec says). Verify `formatMotionNumber` (already exported, `motion-system.ts:53`) covers the `7/12` "fraction" format — it does NOT; it formats a single number. The `X/Y` fraction display needs a custom formatter or two separate count-ups.**

**P1-4 · Adding `premiumMotion.rarity` + `prefersReducedMotion()` to `motion-system.ts` is fine, but `motion-system.ts` is currently pure data with no DOM access.** `prefersReducedMotion()` reads `window.matchMedia` — keep it a function (the spec does, guarded by `typeof window`), never a top-level const, or SSR import in any Server Component breaks. **Action: ensure the export is a function and that no Server Component imports it at module scope.**

**P1-5 · `CardCollectionItem.meta` is `Record<string, unknown> | null` (`repo.ts:176`) — adding `seriesId`/`mascotName`/`mascotColorIdentity` to `meta` needs no migration (good) but every read is `typeof item.meta?.x === "string"` guarded.** The existing code already does this (`questIdFromCard` `:174`, `:750`). **Action: keep the same guarded-read pattern; do NOT tighten the `meta` type to a named interface (it's shared with AI chat meta at `:971`). The `draw/route.ts:95` meta object is the single write site — add the three fields there, sourced from the *card* (deterministic) not the quest, to stay idempotent.**

**P1-6 · Asset-pipeline `gen-mascot-images.mjs` is a faithful clone but has two concrete divergences from the working `gen-market-images.mjs`.** (1) The real script reads key from `process.env.AI_API_KEY` only — but the comment at `gen-market-images.mjs:14` says key may come from `.env.local`; the actual code does **not** load dotenv. The mascot script must likewise be run with `$env:AI_API_KEY` set, or add explicit dotenv loading. (2) `background:"transparent"` is sent to gpt-image-2 via the gpt-agent.cc gateway — **unverified that this gateway honors it** (memory note confirms `b64_json` works for `gpt-image-2`, but says nothing about alpha). **Action: generate ONE mascot first, inspect the alpha channel before generating 14; if the gateway flattens to opaque, fall back to `image_remove_background` or a chroma key in sharp. Budget the @2x variant only for celebration heroes as the spec says (weight discipline is correct, ~40KB target matches the proven market pipeline).**

**P1-7 · `next/image` `object-contain` for transparent mascots vs current `object-cover` — verify no layout regression in `QuestBlindBoxArt`.** Current art uses `object-cover` in a `rounded-[28%]` masked circle (`:548`). Swapping to a transparent `object-contain` cutout (asset-pipeline §6.1) changes the silhouette and removes the circular mask intent. **Action: confirm the visual designer wants the cutout floating (drop-shadow) vs in-circle; they're different looks. The `onError` fallback chain (manifest gate → `mascot-fallback.webp` → existing `quest-cards/front-*.png`) is sound — keep it.**

## P2 — Polish, correctness, and scope-creep guards

**P2-1 · `loading.tsx` skeleton (a11y-spec §5.3) is currently impossible to need — the page is fully SSR/blocking.** `page.tsx` `await`s all data before rendering `StudentQuestDashboard`, so there's no client loading state for the first paint; a `loading.tsx` would only show during the server `await` (navigation). It's still a nice-to-have for slow DB (the CN→us-east-2 5s-timeout noted in memory), but `animate-pulse` must be `motion-safe:animate-pulse` per the spec. **Action: low priority; only add if the route's TTFB is a measured problem.**

**P2-2 · `QuestCardBackArt` "PLEASE SMILE" pastel restyle (integration-map §2.1) — note it's only rendered in the `hidden sm:block` placeholder branch (`:1533`) when no card is drawn yet.** Once a card is collected the back shows `QuestCardArt`, not `QuestCardBackArt`. So the pastel restyle has limited surface and is purely decorative. **Action: low effort, low risk — but don't over-invest; it's a placeholder seen briefly.**

**P2-3 · `QuestCommanderPanel` "convert to mascot 3x3 grid" (integration-map §2.1) is the largest UI rewrite and the least-justified.** It currently drives quest *selection* (`onSelect`, `:574`) — repurposing it to a mascot gallery would break the selection flow the rest of the component depends on (`selectedQuestId`). **Action: descope or keep selection intact and add mascot art as decoration only. Do not remove `onSelect`.**

**P2-4 · Two `role="status"` blocks fire back-to-back on claim (claim→draw chains `:1024`).** a11y-spec §4.1 correctly flags double-announce. The fix (one merged live region, or `aria-live="off"` on `quest-claim-result`) is real and cheap. **Action: make `quest-draw-result` a permanently-mounted `role="status"` container (currently conditionally rendered `:1152`, which can miss first announcement) and suppress the claim announcement.**

**P2-5 · Hardcoded `QuestCardBackArt rarity="common"` (`:1534`) and `QuestCardArt compact` reuse mean the rarity-frame upgrade must thread real rarity through.** The collection grid (`:746`) and drawn-card preview (`:1518`) both render `QuestCardArt` — wrapping each in `QuestCardRarityFrame` (integration-map §2.2-A) touches 3 call sites, not 1. **Action: enumerate all `QuestCardArt`/`QuestCardBackArt` call sites (4 total) when adding the frame wrapper.**

**P2-6 · `computeLearningStreak` + "安心假" (gamification §3) is a new pure-core module replacing the `computeStreak` (net-worth) hero card.** Sound and testable, but it changes `overview` payload shape (`quests.ts:460`) consumed by the hero (`:1088`). It's a `behavior`/`finance` pure-function task with its own `*.test.ts` (Asia/Shanghai day-folding needs a fixed-`now` test). **Action: scope as a standalone lib task; the hero-card swap is a follow-on. Property test the "max 1 gap day doesn't reset" invariant.**

**P2-7 · Color-contrast on new saturated poster backplates (a11y-spec §1.2) is the single most likely AA regression.** The spec's safe-color table is correct and load-bearing — the existing `rarityMeta.epic` already carries a hard-won AA fix comment (`:127-130`). **Action: any new white-on-solid poster MUST use the §1.2 audited backgrounds; saturated orange/amber/yellow/pink get dark `--quest-ink` text, never white. Add these to the `vitest-axe` color-contrast assertions.**

---

### Net verdict

**APPROVE with required changes.** The specs are unusually well-grounded (correct line numbers, real data attributes, accurate token names, faithful pipeline clone). The risk is **not** technical impossibility — it's that the integration-map/asset-pipeline describe the data-model changes as "extensions" when they are coordinated type rewrites across `cards.ts` + `content.ts` (×12) + `quests.ts` (×10) + `draw/route.ts` + `page.tsx`, and they **contradict the motion/a11y specs on `legendary`**. Resolve P0-1 through P0-5 (the `as const` rewrite, the legendary decision, the `drawCard` determinism contract, the dual `StudentQuestItem` build sites, and the focus-trap foundation) as gating prerequisites — these are the items that, if discovered mid-implementation, cause TS-error cascades or silently change every existing student's draw outcome. Everything in P1/P2 is then mechanical.
