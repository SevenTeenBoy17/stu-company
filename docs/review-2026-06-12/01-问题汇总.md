## 二、P0 — 必须在任何对外动作前修复

### SEC-01 邀请码 `ilike` 通配符注入 → 未授权注册成教师 + 批量篡改邀请码
- **文件**：`src/lib/db/repo.ts:887`、`:1002`（注册预订）、`:816`/`:840`（查码）；`src/lib/auth-validation.ts:25`（缺字符白名单）
- **证据**（已主控核验）：
  ```ts
  // repo.ts:882 registerUserByInvite 的原子预订
  const reservedRows = await tx.update(inviteCodes)
    .set({ usesRemaining: sql`${inviteCodes.usesRemaining} - 1` })
    .where(and(
      ilike(inviteCodes.code, input.inviteCode),   // 用户输入直接当 LIKE 模式
      sql`${inviteCodes.usesRemaining} > 0`,
      sql`${inviteCodes.expiresAt} > now()`))
    .returning();
  const invite = toInvite(reservedRows[0]);
  ...
  role: invite.role,                                // 角色取自被匹配到的邀请码
  ```
  ```ts
  // auth-validation.ts:25 仅校验长度，不过滤 % / _
  inviteCode: z.string().min(6, "邀请码至少 6 位。").optional(),
  ```
- **攻击场景**：未认证用户向 `/api/auth/register-by-invite` 提交 `inviteCode="MRB-TE%"`（7 字符，通过校验），`ilike` 匹配种子库的 `MRB-TEACHER-2026` → 自助注册为**教师**，获得班级名册、全部学生 run 数据、布置作业、为学生代付等权限。提交 `"%%%%%%"` 会用一条 `UPDATE` 把**所有**邀请码 `usesRemaining` 同时减一（批量耗尽/篡改），并返回顺序不定的某条角色。
- **修复**：① 邀请码、邮箱等**精确查找**一律改 `eq()`；需大小写不敏感时用 `eq(sql\`lower(${inviteCodes.code})\`, code.toLowerCase())`；② zod 层把 `inviteCode` 限为 `regex(/^[A-Za-z0-9-]+$/)` 白名单。
- **回归测试**：新增用例 `register-by-invite` 提交含 `%`/`_` 的码必须返回"邀请码不存在"，且不得 decrement 任何码。

## 三、P1 — 本轮迭代必修

### SEC-02 认证/账号解析用 `ilike` 做邮箱精确匹配，破坏唯一性
- **文件**：`src/lib/db/repo.ts:621`（`selectUserByEmail`，被登录/查重/找回密码/家庭成员/改邮箱调用）
- **证据**：`.where(ilike(users.email, email)).limit(1)` —— 邮箱本地名合法字符 `_` 是 LIKE 单字符通配符；`limit(1)` 无 `ORDER BY` 返回任意匹配行。
- **影响**：`j_hn@x.com` 匹配 `john@x.com`；注册查重可能误判、找回密码/家庭成员可定位到错误用户，破坏"一邮箱一身份"不变量。
- **修复**：同 SEC-01，改 `eq` 精确比较，规范化邮箱后相等匹配。

### SEC-03 `/api/auth/register-by-invite` 限流键由攻击者控制
- **文件**：`src/app/api/auth/register-by-invite/route.ts:22`
- **证据**：`rateLimit(rateLimitKey("register-invite", body.email.toLowerCase(), request), 5, 600000)` —— 键含请求体 `email`，每次换邮箱即绕过，无 per-IP 预算。
- **影响**：放大 SEC-01，可不受限批量注册教师 / 耗尽邀请码。
- **修复**：对未认证注册类接口按 IP 限流，或叠加 per-IP 失败预算（参照 `/api/auth/login` 双层策略）。

### SEC-04 缺少安全响应头（CSP / HSTS / X-Frame-Options / X-Content-Type-Options）
- **文件**：`next.config.ts`（无 `headers()`）、仓库无 `middleware.ts`
- **影响**：面向未成年人的平台缺纵深防御——无 CSP 时任一 XSS 即可外带；无 frame-ancestors 存在点击劫持面；无 HSTS 有降级风险。
- **修复**：`next.config.ts` 增加 `async headers()`，全路由下发 `Content-Security-Policy`（至少 `default-src 'self'; frame-ancestors 'none'`）、`Strict-Transport-Security`、`X-Content-Type-Options: nosniff`、`Referrer-Policy`。

### DOM-01 确定性升值资产 + 无上限杠杆 = 零风险刷到最高段位（击穿反 YOLO）
- **文件**：`src/lib/simulation.ts:120`（`getPropertyValue` 每回合线性 +2.4%）、`:126`（`getVentureValue` 收官 +36%）、`:366`（`loan` 无上限）
- **证据**：房产/创业估值**不读 seed/eventTimeline**，是与市场无关的确定性升值曲线；债务仅 `×1.018` 复利（`:517`）。
- **作弊场景**：① 开局现金全押房产持有到第 12 回合 → 净值确定 +26%、平滑无回撤 → riskAdj=1.0、drawdown=1.0，配学习满分 `power≈1732`（tier5 财商宗师），全程零判断零风险。② 无上限杠杆：房产 2.4% > 债务 1.8%，借巨款买房产每借 1 元第 12 回合净赚约 0.045 元，纪律地板分(42)抵不过收益 → 仍 tier5。**正面违背 power-score 文档"ranks decision quality, not luck"承诺**——现有 anti-YOLO 测试只防"运气好的赌徒"，没防"确定性套利的死押者"。
- **修复**：① 房产/创业引入与 seed 挂钩的波动/下行；② `loan` 加额度上限（如净资产或本金固定倍数），`leveragePenalty` 随杠杆率非线性增长；③ power-score 增加杠杆率/集中度惩罚项，借入资金购入资产收益打折。

### DOM-02 「什么都不做」基线 1110 战力，纪律分结构性奖励不操作
- **文件**：`src/lib/leaderboard/power-score.ts:53`、`src/lib/simulation.ts:151`（`computeDisciplineScore`）
- **证据**：
  ```ts
  riskAdjReturn: clamp01((riskAdj + 2) / (2 * 2)),  // 0 收益 → 0.5（白送一半）
  drawdown: clamp01(1 - maxDrawdownPct / 50),        // 从不投资 → 永不回撤 → 1.0
  ```

- **修复**：零收益/零波动应给低分而非 0.5；drawdown 满分要求曾承担敞口；纪律分去掉"囤现金 +6"或改为"该出手时出手"的主动性度量；power 要求最少有效回合数。

### DOM-03 战力榜不校验赛季种子、不要求完赛 → 重玩选种 + 见好就收刷榜
- **文件**：`src/lib/leaderboard/service.ts:160`、`src/lib/db/repo.ts:2672`（覆盖式 upsert）/`:2748`（读榜无 seed 过滤）/`:1394`（`replayRunForUser` 随机种子）

### FE-01 沙盘操作按钮防双击失效：`useTransition` 未追踪异步请求
- **文件**：`src/components/student/student-sandbox.tsx:303`（按钮 `disabled={pending}` 行 411/428/593/632/651/682）

- **修复**：`startTransition(async () => { await mutate(url, body).catch(...) })`（与 `wechat-checkout-button.tsx:221` 一致），或自管 `submitting` 在 `finally` 复位。

## 六、做得好的地方（核对不变量后确认成立，勿在修复中破坏）

**安全/鉴权**
- JWT 全部显式 `algorithms:["HS256"]` 校验、7 天 exp，无 alg 混淆/alg=none（`auth.ts:68` 等）。
- Cookie `httpOnly + sameSite:lax + 生产 secure + path:/`（`auth.ts:43`）。
- 服务端会话吊销（tokenVersion）一致生效，登出/改密/改邮箱均 bump。
- `SESSION_SECRET` 生产强制 ≥32（`env.ts:5`）。
- CSRF `checkOrigin()` 覆盖所有 state-changing 路由；`billing/notify` 作为带签名 webhook 不调 checkOrigin 是正确取舍。
- **微信支付链路扎实**：先验 RSA-SHA256 签名后 AES-GCM 解密、缺平台公钥时生产直接拒绝、回调金额与订单强校验、tier 取自订单、按状态幂等——无金额篡改、无重复履约。
