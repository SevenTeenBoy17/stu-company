### 0.2 模型与推理强度
- **高推理强度**（GPT-5-Codex high / 最强档）：Phase 0 编排脚本、Phase 2/3 的 schema+迁移+判分逻辑（容易错、错了难查）。
- **常规强度**：Phase 1 的机械式前端改动（防抖、三态、文案）。
- 一句话：**动 schema / 动钱 / 动判分用高档；改 UI 文案与状态用常规档。**

### 0.3 五条跨 Phase 通用纪律
1. **Plan-first**：任何跨多文件的改动，先让 Codex"只列计划与文件清单、先别改"，你批准后再 apply。提示词模板：`Before editing, list the exact files you will touch and the diff plan. Wait for my "go".`
2. **上下文卫生**：只让 Codex 读"本提示词点名的文件"（PRD/本手册已给精确 `文件:行号`），别让它漫游全仓——上下文越干净越准。**一个 Phase 一个 Codex 会话**。
3. **最小 diff**：禁止顺手重构。提示词固定带一句 `Make the minimal diff that satisfies the steps. Do NOT refactor unrelated code.`（必要时切到 `minimal-change-engineer` 思路。）
4. **客观闸门**：能用 `git grep` 判真假的就别靠主观。每个提示词末尾给一条 grep 验收（如"`void mutate` 必须 0 命中"）。
5. **实现者≠评审者**（AGENTS.md §3.1）：实现用 db_architect/api_wirer/ui_implementer，评审换 reviewer（+按域加 testing-accessibility-auditor / engineering-code-reviewer / education_narrative_designer）。每阶段以 `APPROVE / REQUEST_CHANGES / NEEDS_DISCUSSION` 收尾。

### 0.4 "证明它真能用"的验证纪律（本任务的灵魂）
> 你的核心痛点是"停留于简单交互/点了没反应"。所以**每个 Phase 收尾不止跑 tsc，必须跑真实链路**：
```powershell
npm run lint; if ($?) { npx tsc --noEmit }; if ($?) { npm run test }
npm run db:up        # 库必须 healthy
npm run dev          # 起应用，按该 Phase 的"验收"实际点一遍
```
- 写操作改动 → 必须**在库 Up 时**实际注册/交易/提交一次，刷新后数据还在才算过。
- 失败反馈改动 → **故意把库停掉**（`npm run db:down`）再点一遍，确认显示"中文错误+重试"而非空白/英文码；测完 `npm run db:up` 复原。

### 0.5 提交纪律（AGENTS.md 红线）
- 一个 Phase 一个 PR；`git diff` 自检后**显式 `git add <具体路径>`**，**不要** `git add .`、**不要** `git commit -a`。
- 迁移一律 `npm run db:migrate`（**不要** `drizzle-kit push`，对本库会崩）。
- AI 调用只走 `src/lib/ai.ts`；颜色 token 只在 `globals.css`。

### 0.6 Do / Don't 速查
| Do | Don't |
|---|---|
| Phase 0 先行，全程库 Up | 库停着就改写链路、然后"以为修好了" |
| 每改一处写操作就真点一次+刷新验证 | 只看 tsc 绿就收工 |
| 提示词给精确 `文件:行号` + grep 闸门 | 让 Codex 漫游全仓自己找 |
| 一个 Phase 一个会话/分支/PR | 多个 Phase 混在一个大 diff |
| 失败路径也要测（停库演练） | 只测 happy path |
